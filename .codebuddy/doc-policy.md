# 📋 文档生成政策

## 🎯 核心原则

**✋ 停止自动生成 Markdown 文档**

从现在起，AI 在生成代码和功能实现时：
- ❌ **不再** 自动生成总结、报告类 `.md` 文件
- ❌ **不再** 自动生成过程文档和中间报告
- ✅ **仅** 为必要的功能文档生成 Markdown（如 API 说明、配置指南等）
- ✅ 所有代码生成结束时，输出简洁的文字总结即可

## 📁 文档现状

### 已整理完成
- **docs/guides/** - 15+ 实用指南（日常需要使用）
- **docs/reports/** - 22+ 技术报告（存档参考）
- **docs/specs/** - 8+ 技术规范（存档参考）
- **docs/archives/** - 5+ 历史文档（过时内容）
- **docs/README.md** - 主文档入口
- **docs/INDEX.md** - 完整索引

### 文件总数
- **总计**: ~50 个 Markdown 文件
- **大小**: ~600KB (不应继续增长)

## 🚫 禁止操作

AI **绝对不应该**：
1. 生成新的 `.md` 总结报告文件
2. 生成 `*-SUMMARY.md` 或 `*-REPORT.md` 文件
3. 生成进度、完成、交付类文档
4. 生成 `*-COMPLETE.md` 或 `*-COMPLETION.md` 文件
5. 使用 `write_to_file` 创建大量文本化总结

## ✅ 允许操作

AI **可以**：
1. 更新现有文档（如 `docs/guides/` 中的指南）
2. 为关键功能生成简洁的 API 文档
3. 在代码注释中添加必要说明
4. 在控制台输出文字总结和反馈
5. 在必要时创建配置或模板文件

## 📝 代码完成后的操作

### 之前（不应再做）
```
✗ "现在生成完成报告..." 
✗ write_to_file("IMPLEMENTATION-COMPLETE.md")
✗ 创建 10 个总结文件
```

### 现在（应该做）
```
✓ 在控制台输出简洁文字总结
✓ 执行 git add / git status
✓ 若有关键信息，可更新现有文档（不是新建）
```

## 💡 理由

1. **避免文档爆炸** - 文档已经很多，不需要持续增长
2. **提高效率** - 减少文件 I/O 操作
3. **便于维护** - 现有文档已分类整理，避免重复
4. **用户体验** - 用户需要的是代码和指导，不是报告堆积
5. **代码优先** - 重点应该是功能实现，而非文档生成

## 🗂️ 文档维护流程

### 添加新文档的流程（特殊情况）
1. 必须是**核心功能文档**，不是总结报告
2. 放在 `docs/guides/` 或 `docs/specs/` 中
3. 在 `docs/README.md` 和 `docs/INDEX.md` 中添加链接
4. 保持格式一致，使用现有的文档风格

### 不使用的操作
- ❌ 每次完成任务都生成新的 REPORT 文件
- ❌ 每个 feature 都配一个 SUMMARY 文件
- ❌ 任意位置创建未分类的 markdown 文件

## 🎯 例外情况

**仅在以下情况生成新文档**：
1. **核心 API 文档** - 必须的 API 参考
2. **配置指南** - 新的系统需要配置说明
3. **集成指南** - 新的外部集成文档
4. **故障排除** - 新的已知问题和解决方案

即使在以上情况，也应该：
- 放在相应的 `docs/` 子目录中
- 在 INDEX.md 中添加链接
- 保持文件名规范和格式统一

## ✨ 最终目标

- 📦 文档组织清晰、易于查找
- 📉 停止无限制增长
- ⚡ 专注代码实现和功能开发
- 📌 只在必要时维护和更新文档
