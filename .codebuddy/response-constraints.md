# 🎯 AI 回答约束 Skill - 深圳公园推荐 CLI

这个 Skill 定义了 AI 在本项目中的回答约束和最佳实践。所有代码生成、功能实现、问题解决都应遵循以下规则。

---

## 📋 核心约束

### 1️⃣ 文档生成约束

**严格禁止**：
- ❌ 生成 `*-SUMMARY.md`, `*-REPORT.md`, `*-COMPLETE.md` 等总结报告文件
- ❌ 生成过程文档、中间报告、进度追踪文档
- ❌ 自动创建大量新的 markdown 文件
- ❌ 在根目录创建未分类的 markdown 文件

**允许做**：
- ✅ 更新 `docs/guides/` 中的现有文档
- ✅ 为核心功能创建必要的 API 文档（放在 `docs/specs/`）
- ✅ 在代码中添加注释和文档字符串
- ✅ 在控制台输出文字总结和反馈

**具体例子**：

```
❌ 错误做法：
write_to_file("IMPLEMENTATION-COMPLETE.md", "实现完成报告...")
write_to_file("FEATURE-X-SUMMARY.md", "功能 X 总结...")

✅ 正确做法：
console.log("✅ 功能实现完成\n" + summaryText)
// 更新现有文档：
replace_in_file("docs/guides/FEATURE-GUIDE.md", oldStr, newStr)
```

## 📝 回答内容约束

### 2️⃣ 输出格式要求

**必须遵循**：

1. **简洁性** (CRITICAL)
   - 回答直切要点，避免冗余
   - 代码注释只在必要时添加，不要过度注释
   - 不输出过长的文本总结
   - 目标字数：简明扼要，300 行以内（除非特殊情况）

2. **结构清晰**
   - 使用 markdown 格式清晰分层
   - 重点突出，便于快速理解
   - 复杂概念分步骤说明

3. **避免重复**
   - 不重复说明已在文档中详细描述的内容
   - 直接引用相关文档
   - 提供链接而非全文转述

4. **代码优先**
   - 直接生成可运行的代码
   - 不输出长篇逻辑分析
   - 代码本身就是最好的说明

5. **🚫 禁止生成 MD 文档** (CRITICAL - 每次都要遵守)
   - ❌ **除非用户明确说"生成文档"或"创建 xxx.md"，否则不要生成任何 `.md` 文件**
   - ❌ 不要自动创建总结、报告、完成、进度类的 markdown 文件
   - ❌ 即使完成了复杂任务，也不要主动生成文档
   - ✅ 默认行为：仅在控制台输出简洁的文字总结
   - ✅ 仅在用户显式要求时才生成文档
   - ✅ 所有反馈都通过控制台输出，不写入文件

## 🚀 工作流约束

### 3️⃣ 任务完成流程

**标准流程**（每次都遵守）：

```
1. 分析需求
   ├─ 快速理解用户需求
   └─ 判断：用户是否明确要求生成文档？

2. 实现功能
   ├─ 编写代码（使用 replace_in_file / write_to_file）
   ├─ 执行必要的命令（execute_command）
   └─ 验证功能正常

3. 输出反馈 (默认仅通过控制台)
   ├─ 在控制台打印简洁总结（不生成文件）
   ├─ 说明关键改动点
   └─ 提供必要的后续建议

4. 特殊情况：仅当用户明确要求时
   ├─ 如果用户说"生成文档"、"创建 xxx.md"等
   ├─ 那么才生成相应的文档文件
   └─ 否则永远不生成

5. 清理工作
   ├─ 删除临时文件
   ├─ 提交到 git（如适用）
   └─ 不生成任何未被要求的 md 文档
```

**禁止的流程** (不要做！):

```
❌ 错误 - 永远不要自动做这些:
1. 实现功能完成后，主动生成文档
2. 生成 IMPLEMENTATION-REPORT.md
3. 生成 COMPLETION-SUMMARY.md
4. 生成任何 *-COMPLETE.md 文件
5. 生成任何 *-SUMMARY.md 文件
6. 生成任何 *-REPORT.md 文件

✅ 正确的做法：
- 用户没有说，就不要生成
- 即使完成了很复杂的功能，也只在控制台输出
- 等待用户明确要求："帮我生成文档"、"创建 xxx.md" 等
- 只有这时才调用 write_to_file 生成文档
```

## 📊 代码生成约束

### 4️⃣ 代码质量要求

**必须满足**：

1. **可立即运行**
   - 所有必需的 import 和依赖
   - 正确的配置和路径
   - 完整的错误处理

2. **遵循项目规范**
   - 使用 TypeScript 严格模式
   - 遵循现有代码风格
   - 使用项目中的约定和工具类

3. **添加合理注释**
   - 复杂逻辑才添加注释
   - 不过度注释明显的代码
   - 关键决策点说明理由

4. **包含测试**
   - 关键功能添加单元测试
   - E2E 功能添加集成测试
   - 测试用例清晰有意义

## 💡 特殊情况处理

### 5️⃣ 何时可以创建新文档

**仅限以下情况**（需明确说明理由）：

1. **核心 API 文档** - 如新增外部接口或 SDK
2. **配置指南** - 如新的系统依赖或环境配置
3. **集成指南** - 如新的第三方服务集成
4. **故障排除** - 如新的已知问题和解决方案

**即使创建，也必须**：
- 放在 `docs/` 的适当子目录中
- 在 `docs/README.md` 和 `docs/INDEX.md` 添加链接
- 保持文件名规范 (不用 `*-REPORT.md` 等)
- 通知用户这是必要的新文档

## 📌 回答模板

### 实现功能后的标准回答格式

**重要**: 只在控制台输出，不生成文件！

```
## ✅ 完成

### 📝 改动说明
- 更新了 [文件名]：[简单说明]
- 新增 [功能]：[简洁描述]
- 修复了 [问题]：[简洁原因]

### 🔧 关键改动
[1-3 个要点，每个不超过一行]

### 🚀 使用方式
[简洁的使用说明，如需要]

### 📚 参考文档
[如有相关文档，提供链接]
```

**⚠️ 关键提示**：
- 上面的模板内容应该通过 `console.log()` 或控制台输出
- 不要保存为文件
- 不要创建任何 `.md` 文档
- 即使模板中有 markdown 语法，也只是在控制台中显示，不生成文件

## 🎯 目标与理由

| 约束 | 目标 | 理由 |
|------|------|------|
| 停止生成报告 | 代码优先 | 用户需要功能实现，不是文档堆积 |
| 简洁输出 | 提高效率 | 减少 token 消耗，快速反馈 |
| 代码可运行 | 即用性 | 用户需要立即可用的代码 |
| 遵循规范 | 一致性 | 保持项目代码质量和可维护性 |
| 文档冻结 | 可持续 | 避免无限增长，便于维护 |
